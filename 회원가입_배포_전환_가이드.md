# 교사 회원가입 버전 배포 전환 가이드

이 문서는 이미 Vercel과 Supabase에 배포한 모의 주식 시뮬레이터를 **교사 이메일 회원가입·로그인 버전**으로 바꾸는 설명서입니다. 처음 배포하는 사람도 따라 할 수 있도록 버튼 순서와 확인 방법을 함께 적었습니다.

## 먼저 알아둘 핵심

- 교사는 이메일과 비밀번호로 회원가입하고, 자기 계정이 만든 방만 관리합니다.
- 학생은 별도 이메일 회원가입을 하지 않습니다. 방 PIN, 학번, 이름, 개인 숫자 비밀번호를 그대로 사용합니다.
- 학생 브라우저 내부에서는 Supabase 익명 인증 세션을 자동으로 만듭니다.
- 학생 비밀번호 해시와 접속 세션 토큰은 일반 조회에서 제외됩니다.
- 운영 중인 기존 방에는 교사 계정 소유자 값이 없습니다. 새 권한 SQL을 적용한 뒤에는 기존 방 대신 **로그인한 교사가 새 방을 만들어야 합니다.**

따라서 실제 수업이 진행 중일 때는 전환하지 마세요. 수업이 끝난 뒤 아래 순서대로 작업하세요.

## 전체 순서

```text
진행 중 수업 종료 확인
→ 로컬 lint/build 확인
→ Supabase Auth 설정
→ 회원가입·소유권 마이그레이션 실행
→ 변경 파일을 GitHub main에 업로드
→ Vercel 자동 배포 완료 확인
→ 교사 최초 회원가입과 이메일 확인
→ 새 수업 방 생성
→ 시크릿 창에서 학생 입장·구매·교사 대시보드 확인
```

## 1. 배포 전에 준비하기

다음 계정에 로그인할 수 있어야 합니다.

- 이 앱의 GitHub 저장소
- 이 앱의 Vercel 프로젝트
- 이 앱의 Supabase 프로젝트
- 회원가입 테스트에 사용할 수 있는 교사 이메일

현재 앱 주소와 Supabase 프로젝트 이름이 맞는지 먼저 확인하세요. 다른 프로젝트에서 SQL을 실행하면 다른 앱의 데이터가 바뀔 수 있습니다.

### 진행 중인 수업 확인

1. 운영 앱의 교사 화면을 엽니다.
2. 학생이 아직 접속 중인지 확인합니다.
3. 필요한 최종 보고서와 CSV를 먼저 내려받습니다.
4. 수업이 끝났는지 확인합니다.

새 SQL을 적용한 뒤에는 소유자 값이 없는 예전 방을 새 교사 계정에서 열 수 없습니다. 예전 방을 계속 사용하려 하지 말고, 전환 완료 후 새 방을 만드세요.

## 2. 로컬 파일 검사하기

터미널을 열고 `package.json`이 있는 프로젝트 폴더로 이동합니다.

```bash
cd "모의 주식 시뮬레이터 프로젝트 폴더"
npm install
npm run lint
npm run build
```

정상 결과는 다음과 같습니다.

- `npm run lint`가 오류 없이 끝남
- `npm run build`가 오류 없이 끝남
- 프로젝트 폴더에 `dist`가 생성됨

`dist`와 `node_modules`는 GitHub에 올리지 않습니다.

## 3. Supabase에서 회원가입 기능 켜기

### 3-1. 이메일 로그인 확인

1. [Supabase Dashboard](https://supabase.com/dashboard)에 로그인합니다.
2. 이 앱의 프로젝트를 선택합니다.
3. 왼쪽에서 `Authentication`을 엽니다.
4. `Providers`를 엽니다.
5. `Email` 제공자가 활성화되어 있는지 확인합니다.
6. 이메일 확인 기능을 켜 둡니다.

이메일 확인을 켜면 회원가입 직후 바로 교사 화면이 열리지 않을 수 있습니다. 받은 이메일에서 가입 확인 링크를 누른 뒤 로그인하면 정상입니다.

### 3-2. 학생용 익명 로그인 켜기

1. 같은 `Authentication` 설정에서 `Anonymous Sign-Ins` 또는 익명 로그인 항목을 찾습니다.
2. 익명 로그인을 활성화합니다.
3. `Save`를 누릅니다.

이 설정이 꺼져 있으면 학생 입장 시 `Anonymous sign-ins are disabled`와 비슷한 오류가 표시됩니다. 학생에게 이메일 가입을 시키는 기능이 아니라, 학생 브라우저에 안전한 임시 사용자 ID를 부여하는 기능입니다.

### 3-3. 앱 주소와 이동 주소 등록

1. `Authentication` → `URL Configuration`을 엽니다.
2. `Site URL`에 실제 Vercel 운영 주소를 입력합니다.

```text
https://내-프로젝트.vercel.app
```

3. `Redirect URLs`에 다음 주소를 추가합니다.

```text
http://localhost:5173/**
https://내-프로젝트.vercel.app/**
```

4. 커스텀 도메인이 있다면 그 주소도 추가합니다.
5. `Save`를 누릅니다.

주소 끝의 철자, `https`, 하이픈을 정확히 확인하세요. 이 설정이 틀리면 가입 확인이나 비밀번호 재설정 후 엉뚱한 주소로 이동할 수 있습니다.

### 3-4. 실제 운영 전 이메일 발송 설정

Supabase 기본 이메일 발송은 테스트용 제한이 있습니다. 여러 교사가 가입하는 실제 운영에서는 `Authentication`의 SMTP 설정에 학교 또는 이메일 발송 서비스의 SMTP를 연결하는 편이 안전합니다.

처음 한 계정으로 테스트할 때는 기본 발송으로 시작할 수 있습니다. 메일이 오지 않으면 스팸함을 확인하고, 짧은 시간에 가입·재설정 메일을 반복해서 보내지 마세요.

공식 참고:

- [Supabase 비밀번호 기반 인증](https://supabase.com/docs/guides/auth/passwords)
- [Supabase 익명 로그인](https://supabase.com/docs/guides/auth/auth-anonymous)
- [Supabase RLS](https://supabase.com/docs/guides/database/postgres/row-level-security)

## 4. Supabase 권한 SQL 적용하기

### 기존 프로젝트를 업그레이드하는 경우

기존 운영 프로젝트에는 앞의 두 SQL이 이미 적용되어 있습니다. 아래 **세 번째 파일만** 실행합니다.

```text
supabase/migrations/20260717002707_teacher_auth_and_room_ownership.sql
```

실행 방법:

1. Finder에서 위 파일을 엽니다.
2. 전체 내용을 복사합니다.
3. Supabase Dashboard → `SQL Editor`를 엽니다.
4. `New query`를 누릅니다.
5. 복사한 SQL을 붙여넣습니다.
6. 현재 선택한 Supabase 프로젝트가 맞는지 다시 확인합니다.
7. `Run`을 누릅니다.
8. 성공 메시지가 나오는지 확인합니다.

붉은 오류가 나오면 같은 SQL을 계속 반복 실행하지 말고, 오류 문장을 보관한 뒤 원인을 먼저 확인하세요.

### 새 Supabase 프로젝트를 처음 만드는 경우

다음 세 파일을 순서대로 실행합니다.

```text
1. supabase/migrations/20260615000000_initial_market_class_schema.sql
2. supabase/migrations/20260616120000_savings_forex_triggers.sql
3. supabase/migrations/20260717002707_teacher_auth_and_room_ownership.sql
```

### SQL 적용 후 확인

1. `Table Editor`를 엽니다.
2. `teacher_profiles` 테이블이 보이는지 확인합니다.
3. `rooms`에 `owner_user_id` 열이 있는지 확인합니다.
4. `players`, `student_states`, `final_submissions`에 `user_id` 열이 있는지 확인합니다.

주의: 인터넷에서 본 해결책이라며 모든 테이블을 `anon` 역할에 허용하는 SQL을 실행하면 안 됩니다. 이번 마이그레이션은 입장 전 PIN 미리보기만 익명 키에 허용하고, 실제 테이블은 인증된 세션과 RLS 규칙으로 보호합니다.

## 5. GitHub에 변경 파일 올리기

현재 프로젝트는 로컬 폴더에 `.git`이 없을 수 있으므로 GitHub 웹 업로드 방식으로 설명합니다.

1. GitHub에서 앱 저장소를 엽니다.
2. `Add file` → `Upload files`를 누릅니다.
3. 프로젝트의 변경 파일을 원래와 같은 경로로 업로드합니다.
4. 특히 다음 파일이 빠지지 않았는지 확인합니다.

```text
package.json
package-lock.json
src/App.jsx
src/styles.css
src/lib/supabaseAuth.js
src/lib/supabaseRoomStore.js
supabase/config.toml
supabase/migrations/20260717002707_teacher_auth_and_room_ownership.sql
README.md
회원가입_배포_전환_가이드.md
```

5. `.env`, `.env.local`, `node_modules`, `dist`, `service_role` 키가 업로드 목록에 없는지 확인합니다.
6. 커밋 메시지를 `교사 회원가입과 방 소유권 적용`처럼 입력합니다.
7. `main` 브랜치에 커밋합니다.

GitHub 저장소 첫 화면의 `package.json`에 `@supabase/supabase-js` 의존성이 보이는지도 확인하세요.

## 6. Vercel 배포 확인하기

GitHub와 Vercel이 연결되어 있다면 `main` 커밋 뒤 자동으로 배포가 시작됩니다.

1. [Vercel Dashboard](https://vercel.com/dashboard)를 엽니다.
2. 이 앱의 프로젝트를 선택합니다.
3. `Deployments`를 엽니다.
4. 가장 최근 배포가 `Building`에서 `Ready`로 바뀌는지 확인합니다.
5. 실패했다면 해당 배포를 눌러 Build Logs의 첫 번째 오류를 확인합니다.

환경변수는 다음 두 값이 `Production`에 등록되어 있어야 합니다.

```text
VITE_SUPABASE_URL
VITE_SUPABASE_PUBLISHABLE_KEY
```

Legacy 프로젝트는 두 번째 값 대신 `VITE_SUPABASE_ANON_KEY`도 지원하지만, 새 프로젝트에서는 Publishable key를 권장합니다. `service_role`, `sb_secret_...`, 데이터베이스 비밀번호는 절대로 `VITE_` 변수에 넣지 않습니다.

## 7. 교사 최초 회원가입 테스트

1. Vercel 운영 앱을 엽니다.
2. `교사용 대시보드`를 누릅니다.
3. `회원가입` 탭을 누릅니다.
4. 교사 이름, 이메일, 영문과 숫자를 포함한 8자리 이상 비밀번호를 입력합니다.
5. 비밀번호 확인을 같은 값으로 입력합니다.
6. `회원가입`을 누릅니다.
7. 확인 메일 안내가 나오면 받은 편지함과 스팸함을 확인합니다.
8. 가입 확인 링크를 누릅니다.
9. 앱으로 돌아와 이메일과 비밀번호로 로그인합니다.
10. 오른쪽 위에 교사 이름과 `로그아웃` 버튼이 보이는지 확인합니다.

Supabase `Table Editor` → `teacher_profiles`에 교사 이름이 들어간 행이 하나 생기면 가입 처리가 정상입니다.

## 8. 새 방과 학생 구매 내역 테스트

### 교사 창

1. 로그인한 상태에서 새 수업 방을 만듭니다.
2. 6자리 PIN을 메모합니다.
3. `rooms` 테이블의 새 행에 `owner_user_id` 값이 있는지 확인합니다.

### 학생 시크릿 창

1. 크롬 시크릿 창을 엽니다.
2. 같은 운영 URL을 엽니다.
3. 방 PIN을 입력합니다.
4. 학번, 이름, 숫자 6자리 개인 비밀번호를 입력합니다.
5. 입장한 뒤 교사가 게임과 라운드를 시작합니다.
6. 학생이 주식을 한 종목 구매합니다.

### 다시 교사 창

1. 교사 대시보드의 학생 목록에서 해당 학생을 찾습니다.
2. 학생의 현금, 투자 자산, 총자산이 갱신되는지 확인합니다.
3. 보유 종목에 방금 구매한 종목과 수량이 표시되는지 확인합니다.
4. 학생 창을 새로고침하고 같은 개인 비밀번호로 재입장합니다.
5. 잔고와 보유 종목이 복원되는지 확인합니다.

이 테스트가 통과하면 기존 문제였던 “학생 구매 내용이 교사 대시보드에 보이지 않음”도 데이터 흐름 전체에서 확인한 것입니다.

## 9. 라운드 종료 설명 권장 방식

라운드 종료 설명은 상세하게 제공하는 편이 좋습니다. 다만 처음부터 긴 글을 모두 펼치면 학생이 핵심을 놓칠 수 있으므로 다음 구조를 사용합니다.

1. 먼저 `이번 라운드 핵심 결과`를 2~3줄로 보여줍니다.
2. 각 이슈에 `실제 적용`, `기대감 반영`, `미발생`, `정반대 결과` 배지를 표시합니다.
3. 가격이 움직인 직접 원인을 한 문장으로 설명합니다.
4. `왜 이렇게 움직였나요?` 상세 영역에서 거시 지표와 자산별 영향을 펼쳐봅니다.
5. 마지막에 `반대 결과가 나오려면 어떤 조건이 필요했을까?` 토론 질문을 둡니다.

이 방식은 빠르게 결과만 보려는 학생과 원리를 더 깊게 보려는 학생을 동시에 지원합니다.

## 10. 자주 생기는 오류

### `Anonymous sign-ins are disabled`

Supabase → Authentication 설정에서 Anonymous Sign-Ins를 켜고 저장하세요.

### 가입 메일이 오지 않음

- 이메일 주소 철자를 확인합니다.
- 스팸함을 확인합니다.
- 짧은 시간에 여러 번 요청했다면 잠시 기다립니다.
- 여러 교사가 사용할 운영 환경이면 커스텀 SMTP를 설정합니다.

### 확인 링크를 눌렀는데 localhost로 이동함

Supabase → Authentication → URL Configuration의 Site URL을 Vercel 운영 주소로 바꾸고, Redirect URLs에 운영 주소를 추가하세요.

### `permission denied` 또는 `new row violates row-level security policy`

- 세 번째 마이그레이션이 끝까지 성공했는지 확인합니다.
- 교사는 로그아웃 후 다시 로그인합니다.
- 학생은 시크릿 창을 닫고 새 시크릿 창에서 다시 입장합니다.
- `rooms.owner_user_id`가 현재 교사 계정 값으로 채워진 새 방인지 확인합니다.

### 로그인은 되지만 예전 방이 안 보임

정상 동작입니다. 예전 방은 교사 계정 소유자 정보가 없으므로 새 방을 만들어야 합니다.

### 교사가 학생 화면을 시험한 뒤 로그아웃된 것처럼 보임

한 브라우저에는 한 종류의 Supabase 세션이 유지됩니다. 교사 테스트는 일반 창, 학생 테스트는 시크릿 창 또는 다른 브라우저에서 하세요.
