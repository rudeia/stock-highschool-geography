## 1. 현재 배포 상태

현재 프로젝트는 다음 흐름으로 운영됩니다.

```text
로컬 프로젝트 파일
  └─ GitHub main 브랜치에 업로드
       └─ Vercel이 변경을 자동 감지해 Production 배포
            └─ 빌드된 웹 앱이 Supabase URL과 Publishable key로 접속
                 └─ PostgreSQL 데이터 저장 및 Realtime 변경 구독
```

실제 확인 결과는 다음과 같습니다.

| 항목 | 확인 결과 |
|---|---|
| GitHub | 공개 저장소이며 기본 브랜치는 `main` |
| 최신 원격 커밋 | `66f994ecf9717feaee30aff5a96ed4ab6f65f35e` |
| 로컬과 원격 | 앱 소스·DB 마이그레이션은 동일함. 로컬 `eslint.config.js`에만 백업 폴더 무시 설정이 추가됨 |
| Vercel | GitHub 커밋별 `Production` 배포 기록이 있음 |
| 운영 사이트 | HTTP 200, Vercel 서울 리전 응답 확인 |
| Supabase | 운영 번들에 Project URL과 Publishable key가 정상 포함됨 |
| Supabase 프로젝트 ref | `hyuurhvtvyylnyzrgsue` |
| Data API | 주요 테이블에 Publishable key로 읽기 요청 시 HTTP 200 |
| Realtime | 마이그레이션에서 주요 테이블을 `supabase_realtime` publication에 등록함 |

이 로컬 폴더에는 `.git`과 `.vercel/project.json`이 없습니다. 따라서 현재 로컬 폴더가 Git/Vercel CLI로 직접 연결된 형태는 아닙니다. GitHub 커밋 메시지와 배포 기록상, 파일을 GitHub 웹에서 업로드하고 Vercel Git 연동으로 자동 배포한 방식입니다.

## 2. 준비물

다음 계정과 프로그램이 필요합니다.

- GitHub 계정
- Supabase 계정
- Vercel 계정
- Node.js 20 이상과 npm
- 이 프로젝트 폴더 또는 `배포용_패키지/market-class-personal-deploy` 폴더

배포할 폴더에는 최소한 다음 파일이 있어야 합니다.

```text
.env.example
.gitignore
DEPLOYMENT.md
README.md
eslint.config.js
index.html
package.json
package-lock.json
src/
supabase/
vercel.json
```

`node_modules`, `dist`, `.env`, `.env.local`, `.vercel`, `.vercel-home`은 GitHub에 올리지 않습니다. 현재 `.gitignore`가 이 항목들을 제외합니다.

## 3. 1단계: 로컬 빌드 확인

프로젝트 폴더에서 실행합니다.

```bash
npm ci
npm run lint
npm run build
```

정상이라면 `dist/`가 생성됩니다. `dist/`는 Vercel이 다시 만들므로 GitHub에는 올리지 않습니다.

로컬에서 Supabase까지 연결해 보려면 환경변수 예시를 복사합니다.

```bash
cp .env.example .env.local
```

`.env.local`에는 본인 Supabase 프로젝트 값을 입력합니다.

```env
VITE_SUPABASE_URL=https://본인-project-ref.supabase.co
VITE_SUPABASE_PUBLISHABLE_KEY=본인의-publishable-key
```

기존 Supabase 프로젝트가 Legacy anon key만 제공하는 경우에는 아래 이름도 사용할 수 있습니다.

```env
VITE_SUPABASE_ANON_KEY=본인의-anon-key
```

`VITE_`로 시작하는 값은 빌드 후 브라우저에 공개됩니다. Publishable key 또는 Legacy anon key만 넣고, `service_role` key나 Supabase secret key는 절대로 넣지 않습니다.

## 4. 2단계: Supabase 프로젝트 만들기

1. [Supabase Dashboard](https://supabase.com/dashboard)에서 `New project`를 선택합니다.
2. 조직, 프로젝트 이름, 데이터베이스 비밀번호, 리전을 설정합니다.
3. 프로젝트 생성이 끝나면 `SQL Editor`를 엽니다.
4. 아래 세 파일을 **표시된 순서대로** 각각 새 쿼리에 붙여넣고 실행합니다.

```text
1. supabase/migrations/20260615000000_initial_market_class_schema.sql
2. supabase/migrations/20260616120000_savings_forex_triggers.sql
3. supabase/migrations/20260717002707_teacher_auth_and_room_ownership.sql
```

세 번째 파일을 실행하기 전에 Supabase Authentication에서 Email과 Anonymous Sign-Ins를 활성화하고 Site URL·Redirect URLs를 설정합니다. 기존 운영 앱을 전환할 때는 [회원가입 배포 전환 가이드](./회원가입_배포_전환_가이드.md)를 먼저 읽으세요.

5. `Project Settings` → `API Keys`에서 Publishable key를 확인합니다.
6. `Project Settings` 또는 `Connect` 화면에서 Project URL을 확인합니다.

CLI로 관리하고 싶다면 SQL Editor와 섞어 사용하지 말고, 처음부터 마이그레이션 이력을 기준으로 관리하는 편이 안전합니다.

```bash
npx supabase login
npx supabase link --project-ref 본인-project-ref
npx supabase db push --dry-run
npx supabase db push
npx supabase migration list
```

명령은 버전에 따라 바뀔 수 있으므로 실행 전 `npx supabase <명령> --help`로 현재 옵션을 확인합니다. 공식 참고 자료: [Supabase 데이터베이스 마이그레이션](https://supabase.com/docs/guides/deployment/database-migrations), [Supabase db push](https://supabase.com/docs/reference/cli/supabase-db-push).

## 5. 3단계: GitHub 저장소에 업로드하기

### 현재와 같은 GitHub 웹 업로드 방식

1. GitHub에서 `New repository`를 선택합니다.
2. 저장소 이름을 정하고 공개 또는 비공개 여부를 선택합니다.
3. 빈 저장소를 만든 뒤 `uploading an existing file` 또는 `Add file` → `Upload files`를 선택합니다.
4. 위의 배포 대상 파일과 폴더를 업로드합니다.
5. 커밋 대상 브랜치를 `main`으로 두고 커밋합니다.
6. GitHub 저장소에서 `.env`와 `.env.local`이 올라가지 않았는지 반드시 확인합니다.

### 이후 수정할 때

현재 운영 저장소와 같은 방식이라면 GitHub의 `Add file` → `Upload files`로 바뀐 파일을 다시 올리고 `main`에 커밋합니다. Vercel이 새 커밋을 자동으로 Production에 배포합니다.

파일이 많거나 자주 수정한다면 아래 Git 방식이 더 안전합니다.

```bash
git init
git branch -M main
git add .
git commit -m "Initial deployment"
git remote add origin https://github.com/본인계정/본인저장소.git
git push -u origin main
```

## 6. 4단계: Vercel과 GitHub 연결하기

1. [Vercel Dashboard](https://vercel.com/dashboard)에서 `Add New` → `Project`를 선택합니다.
2. GitHub 계정을 연결하고 방금 만든 저장소의 `Import`를 선택합니다.
3. 프로젝트 루트에 `package.json`이 있다면 Root Directory는 기본값으로 둡니다.
4. 다음 값을 확인합니다.

```text
Framework Preset: Vite
Install Command: npm install
Build Command: npm run build
Output Directory: dist
```

프로젝트의 `vercel.json`에도 같은 설정과 SPA rewrite가 들어 있습니다.

```json
{
  "buildCommand": "npm run build",
  "outputDirectory": "dist",
  "framework": "vite",
  "rewrites": [
    { "source": "/(.*)", "destination": "/index.html" }
  ]
}
```

5. `Environment Variables`에 다음 두 값을 등록합니다.

```text
VITE_SUPABASE_URL
VITE_SUPABASE_PUBLISHABLE_KEY
```

필요하면 `Production`, `Preview`, `Development` 세 환경 모두에 적용합니다. 최소한 실제 운영에 쓰는 `Production`에는 반드시 있어야 합니다.

6. `Deploy`를 누릅니다.
7. 이미 환경변수 없이 배포했다면 변수를 저장한 뒤 `Deployments`에서 최신 배포를 `Redeploy`합니다.

GitHub 연동이 끝나면 `main`에 커밋할 때마다 운영 배포가 갱신되고, 다른 브랜치나 Pull Request는 Preview URL로 배포됩니다. 공식 참고 자료: [Vercel GitHub 배포](https://vercel.com/docs/git/vercel-for-github), [Vercel 환경변수](https://vercel.com/docs/environment-variables).

## 7. 5단계: 배포 후 확인

### 화면 확인

1. Vercel이 제공한 운영 URL을 엽니다.
2. 교사 화면에서 새 방을 만듭니다.
3. 다른 브라우저 또는 시크릿 창에서 방 PIN으로 학생 입장을 시도합니다.
4. 교사 화면에서 라운드 또는 일시정지 상태를 바꿉니다.
5. 학생 화면에 변경 사항이 실시간 반영되는지 확인합니다.
6. 새로고침 후 방과 학생 상태가 복원되는지 확인합니다.

### Vercel 확인

- `Deployments`의 최신 상태가 `Ready`인지 확인합니다.
- 배포의 Source가 GitHub `main`의 최신 커밋인지 확인합니다.
- `Settings` → `Git`에 올바른 저장소가 연결되어 있는지 확인합니다.

### Supabase 확인

- `Table Editor`에서 `rooms`, `players`, `student_states` 등에 테스트 데이터가 생기는지 확인합니다.
- `Database` → `Publications`에서 필요한 테이블이 `supabase_realtime`에 포함됐는지 확인합니다.
- 테스트가 끝나면 테스트 방과 학생 데이터만 정리합니다.

## 8. 업데이트와 장애 대응

### 소스 수정 배포

```text
파일 수정 → 로컬 lint/build → GitHub main 커밋 → Vercel 자동배포 → 운영 확인
```

### DB 구조 수정

1. `npx supabase migration new 변경내용`으로 새 마이그레이션 파일을 만듭니다.
2. 로컬에서 검증합니다.
3. 마이그레이션을 Supabase에 적용합니다.
4. GitHub에 소스와 마이그레이션 파일을 함께 커밋합니다.
5. Vercel 배포 후 전체 흐름을 확인합니다.

애플리케이션이 새 컬럼을 먼저 요구하면 배포 직후 오류가 날 수 있으므로, 호환 가능한 DB 마이그레이션을 먼저 적용한 뒤 프론트엔드를 배포합니다.

### 자주 발생하는 문제

| 증상 | 확인할 곳 |
|---|---|
| Vercel 빌드 실패 | Build Command, Node 버전, `package-lock.json`, 빌드 로그 |
| 새로고침 시 404 | `vercel.json`의 SPA rewrite 포함 여부 |
| Supabase 연결 안 됨 | Vercel 환경변수 이름·적용 환경·Redeploy 여부 |
| 테이블을 찾을 수 없음 | 마이그레이션 3개가 순서대로 실행됐는지 확인 |
| Realtime 미반영 | `supabase_realtime` publication과 브라우저 콘솔 확인 |
| 로컬에서는 되지만 운영에서 안 됨 | Vercel Production 환경변수와 운영 번들 재배포 확인 |

## 9. 현재 프로젝트의 보안 운영

이 프로젝트는 교실 수업용입니다. 교사는 Supabase Auth 이메일 계정을 사용하고, 방 소유권과 학생 익명 사용자 ID를 기준으로 RLS가 적용됩니다.

- 학생 비밀번호 해시와 접속 세션 토큰은 일반 클라이언트 조회 권한에서 제외됩니다.
- 만료 방 정리 함수의 클라이언트 실행 권한은 제거되어 예약 작업에서만 사용합니다.
- 학생 상태, 거래 기록, 제출 자료를 실명이나 민감정보와 함께 저장하지 않는 것이 안전합니다.
- `service_role` key와 Supabase secret key는 GitHub, Vercel의 `VITE_` 환경변수, 브라우저 코드에 절대로 넣지 않습니다.

여러 학교가 사용하는 공개 운영으로 확장할 때는 커스텀 SMTP, 관리자 역할, 감사 로그, 계정 삭제와 개인정보 보존 정책을 추가로 설계해야 합니다. 공식 참고 자료: [Supabase API 보안](https://supabase.com/docs/guides/api/securing-your-api), [Supabase Realtime Postgres Changes](https://supabase.com/docs/guides/realtime/postgres-changes).
