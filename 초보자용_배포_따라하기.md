# 초보자용 모의 주식 시뮬레이터 배포 따라하기

코딩이나 배포를 한 번도 해보지 않은 사람이 처음부터 따라할 수 있도록 작성한 설명서입니다. 한 단계가 끝날 때마다 “성공 확인”을 하고 다음 단계로 넘어가세요.

- 예상 소요 시간: 처음 하는 경우 40~90분
- 비용: GitHub·Supabase·Vercel의 무료 범위로 시작 가능
- 최종 결과: 학생들이 접속할 수 있는 웹 주소 1개
- 기준일: 2026-07-13

## 먼저 전체 구조를 알아두기

이 배포에는 서로 다른 세 가지 서비스가 사용됩니다.

| 서비스 | 하는 일 | 쉽게 비유하면 |
|---|---|---|
| GitHub | 프로그램 소스 파일 보관 | 파일 보관 창고 |
| Supabase | 방·학생·주식·거래 데이터 저장 | 온라인 장부 |
| Vercel | GitHub의 파일을 웹사이트로 변환해 공개 | 웹사이트 출판소 |

실제 흐름은 다음과 같습니다.

```text
내 컴퓨터의 프로젝트 파일
  ↓ GitHub에 올림
GitHub main 브랜치
  ↓ Vercel이 자동 감지
Vercel이 웹사이트를 빌드하고 주소를 만듦
  ↓ 환경변수로 연결
Supabase에서 방·학생·거래 데이터를 읽고 저장
```

## 0단계: 준비물 챙기기

### 0-1. 계정 3개 만들기

다음 사이트에 가입합니다. 가능하면 모두 같은 이메일 계정을 사용하면 관리하기 편합니다.

1. [GitHub](https://github.com)
2. [Supabase](https://supabase.com/dashboard)
3. [Vercel](https://vercel.com)

Vercel은 `Continue with GitHub` 또는 `Sign up with GitHub`를 선택하세요. 그래야 뒤에 GitHub 저장소를 가장 쉽게 연결할 수 있습니다.

### 0-2. 사용할 폴더 선택하기

다음 폴더가 배포용 파일만 모아 둔 곳입니다.

```text
16. 모의 주식 시뮬레이터/
└─ 배포용_패키지/
   └─ market-class-personal-deploy/
```

처음 하는 사람은 `market-class-personal-deploy` 폴더를 사용하는 것이 가장 안전합니다. `node_modules`, 백업, 압축 파일 등이 제외된 배포 전용 복사본입니다.

### 0-3. Node.js 설치 확인

맥의 `터미널` 앱을 엽니다.

1. `Command + Space`를 누릅니다.
2. `터미널`을 입력합니다.
3. 터미널 앱을 실행합니다.
4. 아래 명령을 입력하고 Enter를 누릅니다.

```bash
node --version
```

`v20`, `v22`, `v24` 같은 결과가 보이면 됩니다. `command not found` 라고 나오면 [Node.js 공식 사이트](https://nodejs.org)에서 LTS 버전을 설치한 뒤 터미널을 닫고 다시 엽니다.

## 1단계: 프로젝트가 정상인지 내 컴퓨터에서 확인하기

### 1-1. 터미널에서 프로젝트 폴더 열기

경로를 직접 입력하지 않아도 됩니다.

1. 터미널에 `cd `를 입력합니다. `cd` 뒤에 공백을 하나 넣으세요.
2. Finder에서 `market-class-personal-deploy` 폴더를 터미널 창 안으로 드래그합니다.
3. 터미널에 긴 경로가 자동 입력됩니다.
4. Enter를 누릅니다.

현재 폴더를 확인합니다.

```bash
pwd
ls
```

`package.json`, `src`, `supabase`, `vercel.json` 등이 보이면 올바른 폴더입니다.

### 1-2. 필요한 프로그램 부품 설치

```bash
npm ci
```

몇 분 걸릴 수 있습니다. 글자가 많이 나오는 것은 정상입니다. 마지막에 터미널 입력 표시가 다시 나오고 `ERR!`로 끝나지 않으면 됩니다.

### 1-3. 코드 검사와 빌드

아래 두 명령을 한 줄씩 실행합니다.

```bash
npm run lint
npm run build
```

성공 기준은 다음과 같습니다.

- `npm run lint`: 빨간 오류로 종료되지 않음
- `npm run build`: 마지막에 `built in ...` 또는 `✓ built` 표시
- 프로젝트 폴더에 `dist` 폴더가 생김

`Some chunks are larger than 500 kB`는 오류가 아니라 속도 개선 권고입니다. 현재 프로젝트는 이 경고가 나와도 정상 배포됩니다.

## 2단계: Supabase 데이터베이스 만들기

Supabase는 가능하면 이 모의투자 앱만을 위한 **새 프로젝트**로 만드세요. 다른 앱의 데이터와 섞이지 않아 실수를 줄일 수 있습니다.

### 2-1. 새 프로젝트 생성

1. [Supabase Dashboard](https://supabase.com/dashboard)에 로그인합니다.
2. `New project`를 누릅니다.
3. Organization을 선택합니다. 없다면 무료 Organization을 하나 만듭니다.
4. 다음과 같이 입력합니다.

| 항목 | 예시 | 주의 |
|---|---|---|
| Project name | `market-class-school` | 영문·숫자·하이픈 권장 |
| Database Password | 자동 생성 또는 강한 암호 | 반드시 비밀번호 관리 앱에 저장 |
| Region | 한국과 가까운 리전 | 서울 또는 도쿄 등 |
| Pricing plan | Free | 처음에는 무료 플랜으로 충분 |

5. 프로젝트 생성 화면에 `Automatically expose new tables and functions` 또는 비슷한 항목이 보이면 **체크된 상태**로 두세요.
6. `Create new project`를 누릅니다.
7. 프로젝트 준비가 끝날 때까지 기다립니다. 보통 몇 분 정도 걸립니다.

`Automatically expose...`를 체크하는 이유는 이 앱이 브라우저의 `supabase-js`로 테이블을 읽고 쓰기 때문입니다. 2026년 5월 이후 새 Supabase 프로젝트에서는 이 자동 공개가 꺼져 있을 수 있습니다.

### 2-2. 첫 번째 SQL 실행

SQL은 데이터베이스에 “어떤 표를 만들고 어떻게 사용할지” 알려주는 명령서입니다. 직접 수정하지 말고 파일 전체를 그대로 복사합니다.

1. Supabase 왼쪽 메뉴에서 `SQL Editor`를 누릅니다.
2. `New query` 또는 `New SQL snippet`을 누릅니다.
3. Finder에서 아래 파일을 찾습니다.

```text
market-class-personal-deploy/
└─ supabase/
   └─ migrations/
      └─ 20260615000000_initial_market_class_schema.sql
```

4. VS Code, 텍스트 편집기 등으로 파일을 엽니다.
5. `Command + A`로 전체 선택한 뒤 `Command + C`로 복사합니다.
6. Supabase SQL Editor의 큰 입력창을 한 번 누릅니다.
7. `Command + V`로 붙여넣습니다.
8. 화면 아래 또는 오른쪽의 `Run` 버튼을 누릅니다.
9. `Success. No rows returned` 또는 비슷한 성공 표시가 나올 때까지 기다립니다.

붉은색 Error가 나오면 즉시 두 번째 파일을 실행하지 말고, 오류 메시지를 복사해 보관하세요.

### 2-3. 두 번째 SQL 실행

첫 번째 SQL이 성공한 후 다음 파일을 같은 방식으로 실행합니다.

```text
supabase/migrations/20260616120000_savings_forex_triggers.sql
```

두 번째 파일이 성공한 뒤 다음 세 번째 파일도 같은 방법으로 실행합니다.

```text
supabase/migrations/20260717002707_teacher_auth_and_room_ownership.sql
```

반드시 첫 번째 → 두 번째 → 세 번째 파일 순서로 실행하세요. 세 번째 파일은 교사 회원가입, 교사별 방 소유권, 학생 익명 인증과 안전한 RLS 정책을 적용합니다.

### 2-4. 테이블이 만들어졌는지 확인

1. Supabase 왼쪽 메뉴에서 `Table Editor`를 누릅니다.
2. 아래 이름이 보이는지 확인합니다.

```text
rooms
players
student_states
team_accounts
assets
portfolios
round_events
room_round_results
trade_logs
round_logs
reflections
final_submissions
```

이름이 보이면 SQL이 적용된 것입니다. 현재 행이 0개인 것은 정상입니다. 아직 방을 만들지 않았기 때문입니다.

### 2-5. 회원가입과 학생 익명 인증 켜기

1. Supabase 왼쪽 메뉴에서 `Authentication` → `Providers`를 엽니다.
2. `Email` 로그인이 활성화되어 있는지 확인합니다.
3. `Anonymous Sign-Ins`를 활성화하고 저장합니다.
4. `Authentication` → `URL Configuration`을 엽니다.
5. `Site URL`에 Vercel 운영 주소를 입력합니다. 아직 주소가 없다면 나중에 Vercel 배포 뒤 반드시 다시 설정합니다.
6. `Redirect URLs`에는 아래 두 종류의 주소를 추가합니다.

```text
http://localhost:5173/**
https://내-프로젝트.vercel.app/**
```

모든 테이블을 `anon` 역할에 허용하는 `grant ... to anon` SQL은 실행하지 마세요. 세 번째 마이그레이션이 교사와 학생의 권한을 필요한 범위로 설정합니다.

### 2-6. Project URL과 Publishable key 복사

1. Supabase 프로젝트 상단의 `Connect` 버튼을 누릅니다.
2. Project URL을 찾습니다. 다음과 유사합니다.

```text
https://abcdefghijk.supabase.co
```

3. 임시로 메모장에 복사해 둡니다.
4. `Publishable key`를 찾아 복사합니다. 보통 `sb_publishable_`로 시작합니다.

Connect 화면에서 찾지 못했다면 `Project Settings` → `API Keys` → `Publishable and secret API keys`에서 찾습니다.

다음 키는 절대 선택하지 마세요.

- `Secret key`
- `sb_secret_...`
- `service_role`

이 앱은 브라우저에서 실행되므로 Publishable key만 사용해야 합니다.

## 3단계: Supabase 연결을 로컬에서 테스트하기

이 단계는 생략할 수 있지만, 문제를 미리 찾을 수 있어 권장합니다.

### 3-1. `.env.local` 파일 만들기

터미널이 아직 `market-class-personal-deploy` 폴더를 가리키는 상태에서 실행합니다.

```bash
cp .env.example .env.local
```

Finder에서 숨김 파일이 안 보이면 `Command + Shift + .`을 눌러 숨김 파일 표시를 켭니다.

`.env.local`을 텍스트 편집기로 열고 다음과 같이 입력합니다.

```env
VITE_SUPABASE_URL=https://본인-project-ref.supabase.co
VITE_SUPABASE_PUBLISHABLE_KEY=sb_publishable_본인키
```

주소와 키 앞뒤에 따옴표나 공백을 넣지 마세요. `=` 앞뒤에도 공백을 넣지 않습니다.

### 3-2. 개발 서버 실행

```bash
npm run dev
```

터미널에 다음과 비슷한 주소가 나옵니다.

```text
Local: http://localhost:5173/
```

1. `http://localhost:5173/`를 브라우저에 입력합니다.
2. 앱 화면이 보이는지 확인합니다.
3. `교사용 대시보드` → `회원가입`에서 테스트 교사 계정을 만듭니다.
4. 이메일 확인 링크를 누른 뒤 로그인합니다.
5. 교사 화면에서 테스트 방을 하나 만듭니다.
6. Supabase `Table Editor` → `rooms`에 `owner_user_id`가 들어간 행이 하나 생겼는지 확인합니다.

행이 생겼다면 로컬에서 Supabase까지 정상 연결된 것입니다.

개발 서버를 끄려면 터미널에서 `Control + C`를 누릅니다.

### 3-3. 비밀 파일 점검

`.env.local`은 GitHub에 올리면 안 됩니다. 현재 `.gitignore`에서 제외하지만, GitHub에 올리기 전에 다시 확인하세요.

GitHub에 올릴 파일:

```text
.env.example
.gitignore
DEPLOYMENT.md
README.md
START_HERE.md
eslint.config.js
index.html
package.json
package-lock.json
src/
supabase/
vercel.json
```

GitHub에 올리지 말 파일·폴더:

```text
.env

node_modules/
dist/
.vercel/
.vercel-home/
.DS_Store
```

## 4단계: GitHub에 프로젝트 올리기

여기서는 터미널의 Git 명령을 사용하지 않고, 현재 운영 저장소와 같은 **GitHub 웹 업로드 방식**을 사용합니다.

### 4-1. 빈 저장소 만들기

1. [GitHub](https://github.com)에 로그인합니다.
2. 오른쪽 위 `+` 버튼을 누릅니다.
3. `New repository`를 선택합니다.
4. 다음과 같이 입력합니다.

| 항목 | 예시 |
|---|---|
| Repository name | `market-class-my-school` |
| Description | `교실용 모의 주식 시뮬레이터` |
| Visibility | 코드를 공개해도 되면 Public, 아니면 Private |

5. `Add a README file`, `.gitignore`, License는 체크하지 않습니다. 이미 프로젝트 안에 해당 파일이 있습니다.
6. `Create repository`를 누릅니다.

### 4-2. 파일 업로드

1. 빈 저장소 화면에서 `uploading an existing file`을 누릅니다.
2. Finder에서 `market-class-personal-deploy` 폴더를 엽니다.
3. 폴더 자체가 아니라 **폴더 안의 배포 파일들**을 GitHub 업로드 영역에 끌어다 놓습니다.
4. 업로드 목록에 `.env.local`, `node_modules`, `dist`가 없는지 확인합니다.
5. 화면 아래 `Commit changes` 영역을 찾습니다.
6. 메시지에 `Initial deployment` 또는 `첫 배포 파일 업로드`를 입력합니다.
7. `Commit directly to the main branch`를 선택합니다.
8. `Commit changes`를 누릅니다.

### 4-3. 성공 확인

저장소 첫 화면에 다음이 보이면 됩니다.

- `package.json`
- `vercel.json`
- `src` 폴더
- `supabase` 폴더
- 브랜치 이름 `main`

`market-class-personal-deploy` 폴더가 저장소 가장 밖쪽에 하나 더 생겼다면 파일이 한 단계 깊게 올라간 것입니다. Vercel에서 Root Directory를 그 폴더로 설정하거나, GitHub 저장소 최상위에 `package.json`이 오도록 다시 올립니다.

## 5단계: Vercel에서 GitHub 저장소 배포하기

### 5-1. GitHub 연결

1. [Vercel Dashboard](https://vercel.com/dashboard)에 로그인합니다.
2. `Add New...` → `Project`를 선택합니다.
3. `Import Git Repository` 영역을 찾습니다.
4. GitHub 연결 요청이 나오면 `Continue with GitHub` 또는 `Install Vercel for GitHub`를 선택합니다.
5. 연결 범위를 묻는다면 방금 만든 저장소를 포함합니다.
6. 저장소 이름을 찾아 오른쪽 `Import`를 누릅니다.

저장소가 보이지 않으면 GitHub의 `Settings` → `Applications` → `Installed GitHub Apps` → `Vercel` → `Configure`에서 저장소 접근을 허용합니다.

### 5-2. 빌드 설정 확인

Vercel이 `package.json`과 `vercel.json`을 읽으면 대부분 자동으로 설정합니다. 배포 전에 다음과 같은지 확인하세요.

```text
Framework Preset: Vite
Root Directory: ./
Install Command: npm install
Build Command: npm run build
Output Directory: dist
```

`Root Directory`는 GitHub 저장소 첫 화면에 `package.json`이 보이면 `./` 또는 기본값입니다.

### 5-3. Supabase 환경변수 등록

환경변수는 웹사이트가 어느 Supabase 프로젝트에 접속할지 알려주는 설정입니다.

1. Vercel의 `Environment Variables` 영역을 펼쳐 엽니다.
2. 첫 번째 변수를 등록합니다.

| 칸 | 입력값 |
|---|---|
| Name | `VITE_SUPABASE_URL` |
| Value | Supabase에서 복사한 `https://....supabase.co` 주소 |

3. `Add` 또는 `Save`를 누릅니다.
4. 두 번째 변수를 등록합니다.

| 칸 | 입력값 |
|---|---|
| Name | `VITE_SUPABASE_PUBLISHABLE_KEY` |
| Value | Supabase에서 복사한 `sb_publishable_...` 키 |

5. 적용 환경을 선택하는 부분이 있다면 `Production`, `Preview`, `Development` 모두 체크합니다.

변수 이름은 대소문자, 밑줄, `VITE_`를 포함해 정확히 같아야 합니다. 변수 이름에는 공백이 없습니다.

### 5-4. 첫 배포

1. `Deploy`를 누릅니다.
2. `Building`, `Deploying` 등의 표시가 지나갈 때까지 기다립니다.
3. `Congratulations!`, `Ready` 또는 비슷한 성공 화면이 나오면 배포가 끝난 것입니다.
4. `Visit` 또는 배포 URL을 누릅니다.

주소는 다음과 비슷합니다.

```text
https://market-class-my-school.vercel.app
```

### 5-5. 환경변수를 나중에 넣은 경우

이미 배포한 후 환경변수를 넣었다면 자동으로 기존 배포에 반영되지 않습니다.

1. Vercel 프로젝트 → `Deployments`로 갑니다.
2. 가장 위 배포 오른쪽 `...`를 누릅니다.
3. `Redeploy`를 선택합니다.
4. 확인 창에서 다시 `Redeploy`를 누릅니다.

## 6단계: 진짜로 작동하는지 테스트하기

화면이 열린다고 배포가 완전히 성공한 것은 아닙니다. Vercel 화면, Supabase 저장, Realtime을 모두 확인합니다.

### 6-1. 기본 화면

1. Vercel 운영 URL을 크롬에서 엽니다.
2. 새로고침합니다.
3. 404 페이지가 아니라 모의 주식 시뮬레이터 화면이 보이는지 확인합니다.

### 6-2. 교사 방 생성

1. `교사용 대시보드` → `회원가입`에서 교사 계정을 만듭니다.
2. 확인 메일의 링크를 누른 뒤 로그인합니다.
3. 테스트 방을 만듭니다.
4. 6자리 PIN 번호가 생성되는지 확인합니다.
5. Supabase → `Table Editor` → `rooms`를 엽니다.
6. 방금 만든 방의 PIN과 `owner_user_id`가 보이는지 확인합니다.

이 행이 있으면 Vercel → Supabase 저장이 성공한 것입니다.

### 6-3. 학생 접속

1. 시크릿 창을 엽니다. 크롬은 `Command + Shift + N`입니다.
2. 같은 Vercel 운영 URL을 엽니다.
3. 학생 입장 화면에서 방 PIN, 번호, 닉네임 등을 입력합니다.
4. 입장합니다.
5. 교사 화면에 학생이 보이는지 확인합니다.
6. Supabase `players` 또는 `student_states` 테이블에 학생 행이 생겼는지 확인합니다.

### 6-4. Realtime 테스트

1. 교사 화면에서 라운드, 장 상태, 일시정지 중 하나를 바꿉니다.
2. 학생 화면을 새로고침하지 않고 변경이 반영되는지 보세요.
3. 학생 화면을 새로고침해도 방 상태가 복원되는지 확인합니다.

이 테스트까지 통과하면 배포가 완료된 것입니다.

## 7단계: 나중에 수정한 내용 배포하기

현재 방식을 그대로 따라하면 GitHub의 `main`에 수정 파일을 올릴 때 Vercel이 자동 배포합니다.

1. 로컬에서 파일을 수정합니다.
2. `npm run lint`를 실행합니다.
3. `npm run build`를 실행합니다.
4. GitHub 저장소에서 `Add file` → `Upload files`를 누릅니다.
5. 바뀐 파일을 동일한 경로에 올립니다.
6. `Commit directly to main`을 선택하고 커밋합니다.
7. Vercel → `Deployments`에서 새 배포가 `Building` → `Ready`로 바뀌는지 확인합니다.
8. 운영 URL을 열고 수정 내용을 확인합니다.

GitHub의 다른 브랜치에 올리면 Preview 배포가 만들어질 수 있고, `main`에 올리면 Production 배포가 갱신됩니다.

## 8단계: 오류가 났을 때 해결하기

### 오류 1. `npm: command not found`

원인: Node.js가 설치되지 않았거나 터미널이 설치 전부터 열려 있습니다.

해결:

1. Node.js LTS를 설치합니다.
2. 터미널을 완전히 닫고 다시 엽니다.
3. `node --version`, `npm --version`을 확인합니다.

### 오류 2. `Missing script: build`

원인: 터미널이 `package.json`이 없는 다른 폴더를 가리키고 있습니다.

해결:

```bash
pwd
ls
```

`package.json`이 보이는 폴더로 다시 이동합니다.

### 오류 3. Vercel에서 `No Output Directory named dist found`

원인: Build Command 또는 Root Directory가 틀렸습니다.

해결:

- Root Directory에 `package.json`이 있는지 확인
- Build Command: `npm run build`
- Output Directory: `dist`
- Framework Preset: `Vite`

### 오류 4. 사이트는 열리지만 방이 생성되지 않음

가장 흔한 원인은 환경변수입니다.

1. Vercel → Project → `Settings` → `Environment Variables`로 갑니다.
2. `VITE_SUPABASE_URL`이 정확한지 확인합니다.
3. `VITE_SUPABASE_PUBLISHABLE_KEY`가 정확한지 확인합니다.
4. 두 변수의 환경에 `Production`이 포함됐는지 확인합니다.
5. 저장한 후 반드시 `Redeploy`합니다.

### 오류 5. `permission denied` 또는 RLS 정책 오류

원인: 세 번째 회원가입·권한 마이그레이션이 빠졌거나, 로그인하지 않은 사용자가 테이블에 접근했습니다.

해결:

1. 세 번째 SQL이 성공했는지 확인합니다.
2. 교사는 로그아웃한 뒤 다시 로그인합니다.
3. 학생은 새 시크릿 창에서 PIN과 개인 비밀번호로 다시 입장합니다.
4. 모든 테이블을 `anon`에 허용하는 SQL로 우회하지 마세요.

### 오류 6. `relation public.rooms does not exist`

원인: 첫 번째 마이그레이션 SQL이 실행되지 않았거나 다른 Supabase 프로젝트에서 실행했습니다.

해결:

1. Supabase 상단의 프로젝트 이름을 확인합니다.
2. 첫 번째 SQL을 다시 실행합니다.
3. 성공 후 두 번째 SQL을 실행합니다.

SQL은 `if not exists`를 많이 사용하므로 같은 프로젝트에서 다시 실행해도 대부분 기존 테이블을 중복 생성하지 않습니다. 그래도 붉은 오류 메시지가 나오면 반복 실행하지 말고 오류 내용을 먼저 확인하세요.

### 오류 7. 개별 주소는 열리지만 새로고침하면 404

원인: SPA rewrite 설정이 빠졌습니다.

해결: GitHub 저장소 최상위에 `vercel.json`이 있고 다음 rewrite가 포함됐는지 확인합니다.

```json
"rewrites": [
  { "source": "/(.*)", "destination": "/index.html" }
]
```

### 오류 8. GitHub에 올렸는데 Vercel 배포가 시작되지 않음

1. GitHub 저장소의 현재 브랜치가 `main`인지 확인합니다.
2. Vercel → Project → `Settings` → `Git`로 갑니다.
3. Connected Git Repository가 방금 만든 GitHub 저장소인지 확인합니다.
4. Production Branch가 `main`인지 확인합니다.

## 9단계: 반드시 알아둘 보안 주의사항

이 프로젝트는 교실 수업용 교육 앱입니다.

- 교사는 Supabase Auth 이메일 계정으로 로그인합니다.
- 방 소유권과 학생 익명 사용자 ID를 기준으로 RLS가 적용됩니다.
- 학생 비밀번호 해시와 접속 세션 토큰은 일반 조회에서 제외됩니다.
- 학생 실명, 전화번호, 이메일, 주소, 민감한 상담 정보를 저장하지 마세요.
- `Secret key`, `service_role`, 데이터베이스 비밀번호를 GitHub에 올리지 마세요.
- Vercel의 `VITE_` 변수는 브라우저 번들에 포함되므로 Publishable key만 넣으세요.

학교 밖의 불특정 다수가 사용하는 공개 서비스로 확장하려면 커스텀 SMTP, 관리자 권한, 감사 로그, 계정 삭제와 개인정보 보존 정책을 추가로 설계하세요.

## 용어 사전

| 용어 | 뜻 |
|---|---|
| Repository | GitHub에서 프로젝트 파일을 보관하는 곳 |
| Commit | 현재 파일 상태를 GitHub에 하나의 기록으로 저장하는 것 |
| main branch | 운영 배포의 기준이 되는 기본 작업 줄 |
| Build | React 소스를 브라우저가 읽을 수 있는 파일로 변환하는 과정 |
| Deploy | 빌드한 웹사이트를 인터넷에 공개하는 것 |
| Production | 실제 사용자가 접속하는 운영 배포 |
| Preview | 운영에 반영하기 전 미리 확인하는 배포 |
| Environment Variable | 코드 밖에서 URL·키 등을 전달하는 설정값 |
| Migration | 데이터베이스 구조를 만들거나 바꾸는 SQL 파일 |
| RLS | 사용자가 어떤 행을 읽고 수정할 수 있는지 정하는 데이터베이스 규칙 |
| Realtime | 새로고침 없이 데이터 변경을 다른 화면에 전달하는 기능 |
| Publishable key | 브라우저에서 Supabase 프로젝트를 식별하는 공개용 키 |
